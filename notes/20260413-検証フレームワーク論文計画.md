## 1. ParaStell が原理的に手の届かない問題マップ

「実装ギャップ」(時間と労力で埋められる) と「原理的ギャップ」(設計思想の制約) を区別する。論文の novelty は後者に宿る。本計画は A を直接ターゲットとするが、B-H は中長期の論点リストとして残しておく価値がある。

### A. 検証ギャップ (verification gap) — 採用案 1 のターゲット

**問題**: 中性子輸送結果 (TBR, 壁負荷, 遮蔽性能) の妥当性が、CAD 表現方式に依存する系統誤差を含む可能性がある。statistical error やメッシュ収束は検証されているが、**CAD 表現由来の系統誤差は誰も定量化していない**。文献検索でも 2025 年時点で stellarator geometric UQ の specific paper は見当たらない (核データ UQ の paper はある)。

**ParaStell が手の届かない理由**: ParaStell は単一の CAD 表現方式 (CadQuery loft) しか持っていないので、自分自身でクロス検証ができない。「同じ ParaStell で 2 回計算しても同じ結果」=「再現性はあるが妥当性は別」という構造的盲点。

**cadrum の意義**: 独立な CAD 表現 (Interpolate ベースのテンソル積 B-spline) を提供する verification toolchain になる。これは ParaStell の競合ではなく **補完者** として位置付けられる。

### B. 周期構造の構造的表現

**問題**: stellarator は field period $N$-fold 対称。ParaStell は 1 周期分を作って rotate + fuse で全周にする。fuse は OCCT boolean なので C⁰ 継ぎ目を生む。境界の数値ノイズと中性子 leak リスク。

**ParaStell が手の届かない理由**: CadQuery `makeLoft` が `closed=True` フラグを公開していない。OCCT の IsSame trick はラッパー越しに使えない。

**cadrum の意義**: `Solid::loft(sections, closed=true)` 1 行で C² 連続の閉曲面。Interpolate にすればさらに `SetVPeriodic` で field period をそのまま焼き込める。

### C. 安定 UV パラメータと physics-engineering 結合

**問題**: 中性子壁負荷, 熱流束, 粒子フラックスといった物理量は CAD 表面上に貼られる必要がある。EMC3-EIRENE, BOUT++ の出力を「FW のここに何 MW/m²」とマッピングしたい。これには **CAD 表面に安定した UV パラメータ**が要る。

**ParaStell が手の届かない理由**: ThruSections は断面ごとの knot を互換化するため、loft 後の UV 座標は元の VMEC $(\theta, \phi)$ と対応しない。物理量を CAD に貼る経路が **曖昧**になる。

**cadrum の意義**: テンソル積 B-spline は **入力グリッドの $(\theta, \phi)$ がそのまま $(u, v)$ パラメータ**になる。VMEC 物理座標と CAD UV 座標が同型になる。physics-engineering coupling pipeline の基盤になる。

### D. 微分可能 CAD と inverse design

**問題**: 現代の stellarator 設計は SIMSOPT, DESC など **自動微分ベースの最適化フレームワーク**で動いている (DESC は MHD 平衡を end-to-end 微分可能に)。だが「physics → CAD → neutronics → TBR」の chain で、CAD 部分が微分可能でないため **TBR を目的関数とする adjoint 最適化が成立しない**。

**ParaStell が手の届かない理由**: boolean op は微分不可能。ThruSections の knot 互換化も discrete な処理を含む。

**cadrum の意義**: テンソル積 B-spline は制御点に対して線形なので、**$\partial S(u,v) / \partial P_{ij}$ が解析的に書ける**。aerospace 分野では CAD-based adjoint shape optimization が確立しているが、fusion blanket 文脈には移植されていない。

### E. トポロジー柔軟性: field period 数の比較

**問題**: 4-period vs 5-period vs 6-period の stellarator design を体系的に比較したい。現状 ParaStell では rib 構造が period 数で変わるので比較困難。

**cadrum の意義**: パラメータ化された $(u, v)$ B-spline 表面なら period 数を引数にできる。

### F. 時間発展シナリオへの結合

**問題**: プラズマ scenario が時間発展すると LCFS が動く。**scenario robustness** (どの scenario でも遮蔽が成立するか) を評価したい。ParaStell は static CAD なので追従できない。

**cadrum の意義**: $(u, v, t)$ の **3D テンソル積 B-spline** を作れば、時間方向に補間された surface family を 1 つの object として扱える。文献に類例なし。

### G. SOL/divertor 領域

**問題**: ParaStell は LCFS (閉曲面) のみ。SOL は open field line で「閉じた surface」ではない。divertor 設計は SOL physics と表面形状の coupling が本質。

**cadrum の意義**: SOL の流線追跡データから open surface (free-form B-spline patch) を作る経路を提供できる。論文 1 本分以上のスコープ。

### H. 多物理 mesh 一貫性

**問題**: 中性子, 熱, 構造, EM, CFD は同じ CAD だが異なる解像度・特徴保持ルールを要求する。

**cadrum の意義**: STEP 出力経由で commercial mesher に流せる。pragmatic な貢献。

## 3. 採用案 (案 1) の段階的実行計画

OpenMC 立ち上げが含まれるので **3 つの phase に分け、各 phase 終了時点で publishable な単位**を作る。

### 重要な再フレーミング: 2-方式比較ではなく「収束研究」として実施

当初想定の「ParaStell loft vs cadrum Interpolate の Hausdorff 比較」だけだと、ParaStell が思いのほか正確 (例: < 0.1 mm) だった場合に論文の主張が崩れる risk がある。これを避けるため、実験設計を **section 数 $N$ を変えた収束研究**に再フレーミングする:

- **横軸**: ParaStell の rib (section) 数 $N \in \{5, 10, 20, 40, 80\}$
- **縦軸 (4 種)**:
  - Hausdorff 距離 ($C^0$ メトリクス)
  - 法線角偏差の最大値・平均 ($C^1$ メトリクス)
  - 曲率偏差 ($C^2$ メトリクス)
  - TBR (Phase 1-2 で追加)
- **参照解**: cadrum の Interpolate (input grid 密度を上げた収束済み解) を ground truth とする

**この設計の利点**:

1. **どんな結果でも論文になる**: ParaStell が早く収束すれば「ParaStell loft is verified」という positive verification、収束が遅ければ「verification gap exists」という negative result。両方とも knowledge contribution
2. **多次元メトリクス**: Hausdorff だけでなく $C^1, C^2$ の連続性メトリクスを並べることで、「位置は収束しているが法線が振動している」のような subtle な失敗を検出できる
3. **実用設定の妥当性検証**: ParaStell ユーザーが実際に使う section 数 (典型 10-20) が収束しているかを検証 — それ自体が community に valuable
4. **論文タイトルが立てやすい**: *"Convergence study of section-based stellarator CAD pipelines"* など

### 重要: なぜ Hausdorff 距離だけでは不十分か

中性子輸送計算が見るのは「点の位置」そのものではなく、以下を経由する:

1. **層通過経路長**: 入射角 $\theta$ で層厚 $t$ を通る距離は $t / \cos\theta$。grazing angle ($\theta \to 90°$) では分母がゼロに近づき、わずかな**法線方向誤差が経路長を増幅**する
2. **入射角分布**: 反射・透過確率は $\cos\theta$ 依存。surface normal の不連続 ($C^1$ 欠陥) は局所的に入射角を飛ばす
3. **tally cell binning**: CAD 表面に bind された tally bin が curvature 不連続点で歪む
4. **Lost particle rate**: 数値的不連続点で粒子が「迷子」になる — 数値健康指標として観測可能

つまり **Hausdorff < 0.1 mm でも、$C^1, C^2$ 不連続経由で TBR の local 分布や peak 値に効く可能性は十分にある**。Phase 0 ではこれらを並走計測する。

### Phase 0: CAD 単独検証 MVP (1-2 ヶ月)

**目的**: 中性子計算抜きで「2 つの CAD 表現方式の幾何的差」を定量化する。OpenMC 立ち上げと並走できるので、リスクヘッジとして最初に置く。

**成果物**:
- `Solid::interpolate_surface(grid, periodic_u, periodic_v)` 新 API
- W7-X 公開 VMEC データから平衡面を 2 通り生成する example
- ParaStell loft 出力との Hausdorff 距離・曲率不連続マップ
- (この段階で書けるなら) "geometric verification of stellarator CAD pipelines" 短報

**実装ステップ**:

1. **Interpolate API の FFI 公開** (1-2 週)
   - `cpp/wrapper.h/cpp` に `make_bspline_surface_from_grid` を追加
   - `GeomAPI_PointsToBSplineSurface::Interpolate(pts)` を呼び、`Approx_IsoParametric` か `Approx_ChordLength` を grid 性質で選択
   - `BRepBuilderAPI_MakeFace` → `Sewing` → `MakeSolid` で Solid 化
   - 周期 flag 対応: `surf->SetUPeriodic()` / `SetVPeriodic()`
   - 既存パターン (`make_loft`, `make_gordon_direct`) を踏襲

2. **Rust 側 API 統合** (1 週)
   - `src/traits.rs` に `fn interpolate_surface(grid, ...) -> Result<Solid>` を追加
   - `build_delegation.rs` 経由で自動 delegation 生成
   - `examples/11_interpolate_torus.rs` で smoke test (08, 09 と並行する位置付け)

3. **VMEC データ読み込み** (1-2 週)
   - W7-X 公開ベンチマーク (PPPL, IPP) の wout NetCDF ファイルを入手
   - 最小限の VMEC reader を `src/io/vmec.rs` に追加 (NetCDF 依存は重いので、boundary fourier coef 経由の ASCII 読み込みも検討)
   - フーリエ係数 → $(R, Z, \phi)$ → 3D 点グリッドへの展開

4. **ParaStell との直接比較** (2-3 週)
   - ParaStell をローカルに install (CadQuery 環境)
   - 同じ VMEC 点群から ParaStell loft surface を生成 (STEP 出力)
   - cadrum 側は Interpolate で同じ点群から surface を生成
   - 両 STEP を OCCT ベースの距離計測 (`BRepExtrema_DistShapeShape`) で比較
   - Hausdorff 距離の最大値・分布を出力

**Phase 0 で書ける論文形式**: short paper / letter として "ステラレーター CAD 表現の幾何的差は最大 X mm に達する。これは TBR に effect Y を与え得る (まだ未検証)" と問題提起する形。Frontiers in Nuclear Engineering の short communication, FED の technical note などが候補。

**Phase 0 のリスク**:
- ParaStell の install: CadQuery のバージョン依存が厄介。Docker 化されていれば楽
- VMEC データの形式不整合: PPPL の wout file を直接読むのは boilerplate が多い。代替として `simsopt` か `pyVMEC` の Python ラッパーで前処理して JSON にダンプする経路もあり
- **null result risk** (ParaStell が想定外に正確だった場合): 下記「シナリオ別 pivot 戦略」参照

### Phase 0 のシナリオ別 pivot 戦略

Phase 0 の実験結果が想定通りにならなかった場合のフォールバック:

**S1: 距離 0.1mm 以上、TBR 差も統計誤差以上 (ベストケース)**
- 当初想定通り。論文は「verification gap 確認」の明確な結論
- そのまま Phase 1, 2 に進む

**S2: 距離 0.1mm 未満、しかし $C^1/C^2$ メトリクスでは差あり**
- *準ベストケース。むしろ contribution として novel*
- 論文の主張: 「verification gap は位置ではなく higher-order properties に潜む。Hausdorff だけ見るのは不十分」
- これは fusion CAD コミュニティに対する強いメッセージで、メソッド論として独立価値あり
- そのまま Phase 1, 2 に進み、$C^1$ 差が TBR local 分布に効くかを確認

**S3: 距離も higher-order も差あり、しかし TBR 積分量は差なし**
- *「verification 文化を肯定する条件付き結果」*
- 論文の主張: 「TBR (積分量) は CAD ノイズに対して robust だが、local 量は CAD 依存」
- Phase 2 で **local tally** (壁負荷分布、遮蔽最薄部 leak、divertor target 熱流束) に focus shift
- ParaStell 派にとって「まあ TBR は信用していい」という reassurance を提供しつつ、「local 量を信用するときは要注意」という caveat を提示

**S4: すべて差なし (ワーストケース)**
- *Phase 0 結果が出た時点で early pivot*
- 選択肢 a: **収束研究の極端化** — section 数 $N \in \{3, 5, 8\}$ など実用 coarse 設定でも検証し、coarse での妥当性を確認 (これでも十分論文)
- 選択肢 b: **より過激な形状** — HSX, precise QA, 高 helical excursion な configuration に切り替える。stellarator 群の中で「ThruSections の限界が出やすい形状」を探す
- 選択肢 c: **意図的な"悪い"設定** — ParaStell の rib 間隔を意図的に粗くして verification の感度を上げる
- 選択肢 d: **方向転換** — 案 3 (UV coupling) または案 4 (review) に pivot。案 3 は Interpolate API がそのまま使えるので投資回収可能

S4 の場合の論文ドラフトの最低保証: 「ParaStell loft は実用 section 密度で十分 verified である」という結論は、それ自体 community に valuable な確証であり、tool paper として publish 可能。novelty は薄いが waste of work ではない。

### Phase 1: OpenMC + DAGMC 立ち上げ (2-4 ヶ月)

**目的**: 中性子輸送計算環境を最小構成で立ち上げ、cadrum と ParaStell の Solid を **DAGMC 経由で OpenMC に渡す pipeline** を作る。

**成果物**:
- OpenMC + DAGMC 環境 (Docker 化推奨)
- 簡易ブランケットモデル (FW + breeder layer + shield) を生成する Python スクリプト
- cadrum / ParaStell どちらの平衡面でも同じ層構造を被せられる helper

**実装ステップ**:

1. **OpenMC + DAGMC 環境構築** (2-3 週)
   - Dockerfile or conda env で OpenMC と DAGMC をビルド
   - 最小 example (簡単なトカマク模擬体系で TBR が出ること) で動作確認
   - 公式チュートリアル (OpenMC + DAGMC for fusion) を写経

2. **CAD → DAGMC 変換** (1-2 週)
   - cadrum の STEP 出力 → MOAB → DAGMC h5m への変換
   - ParaStell は標準で h5m 出力を持つので参照
   - メッシュ品質のチェック (lost particles 0 を目標)

3. **層構造の追加** (2-3 週)
   - 平衡面 → offset → ブランケット内側面 → 別 offset → 外側面
   - cadrum 側で `outer.cut(inner)` を呼ぶ層生成スクリプト
   - 材料定義: Li-6 富化 PbLi or LiPb 等の標準 breeder 材

### Phase 2: TBR クロス計算と論文 (2-4 ヶ月)

**目的**: 同じ層構造設定で cadrum / ParaStell 両方の TBR を計算し、差を統計誤差と比較する。

**成果物**:
- W7-X benchmark の TBR 表 (両方式)
- 統計誤差と CAD 差の相関プロット
- 断面密度収束テスト (rib 数を 2x, 4x に振る)
- 論文ドラフト

**実装ステップ**:

1. **TBR 計算** (2-3 週)
   - 両方式で OpenMC を回す (各 $10^7$ ヒストリ程度)
   - tally 設定: TBR, 中性子壁負荷分布, ブランケット heating
   - 差分の統計検定 ($\chi^2$ テスト)

2. **収束テスト** (1-2 週)
   - rib 数を 1x, 2x, 4x に振り、TBR 収束を見る
   - cadrum (Interpolate) は格子密度を変えて並行テスト
   - 「ParaStell は収束していないが cadrum は収束している」が示せれば論文の strongest point

3. **論文ドラフト** (1-2 ヶ月)
   - 序論: notes/20260413-CAD表現方式と中性子輸送系統誤差.md の内容をベース
   - Methods: cadrum の Interpolate API, ParaStell loft, 比較メトリクス
   - Results: Hausdorff, TBR 差, 収束テスト
   - Discussion: verification gap という概念のフレーミング, 他の fusion CAD ツールへの含意

## 5. Verification (Phase 0 終了時点)

Phase 0 が完成したかどうかを確認する end-to-end テスト:

### 機能的検証

1. **解析的テスト**: トーラス ($r=1, R=3$) の解析 grid から `Solid::interpolate_surface` で Solid を生成し、体積が $2\pi^2 R r^2$ と一致する (許容誤差 < 0.1%)
2. **球テスト**: 球面 grid (極退化あり) から Solid を生成し、`MakeSphere` の体積と一致する
3. **W7-X smoke test**: 公開 VMEC データを読み込み、生成された surface が visible (SVG/STEP 出力で確認)
4. **Rust テスト**: `tests/interpolate_surface.rs` で上記 1, 2 を `cargo test` で自動化

### 実験的検証 (収束研究の出力)

5. **section 収束プロット**: ParaStell 側で $N \in \{5, 10, 20, 40, 80\}$ で loft を生成し、$N=80$ を ground truth として Hausdorff, 法線角偏差, 曲率偏差を $N$ の関数としてプロット
6. **cadrum との比較**: 同じ point grid から Interpolate で生成した surface を独立 ground truth として、ParaStell 各 $N$ との距離を計測
7. **不連続の局在**: $C^1$ 不連続点 (= section 境界) が surface 上のどこにあるか、法線角偏差の局所マップを SVG で可視化
8. **シナリオ判定**: 結果が S1, S2, S3, S4 のどれに該当するかを判定し、Phase 1 への進路を決定
9. **ノート更新**: `notes/20260413-CAD表現方式と中性子輸送系統誤差.md` に Phase 0 の数値結果と判定したシナリオを追記

End-to-end の最終確認は `cargo run --example 13_parastell_comparison -- W7X.wout --section-counts 5,10,20,40,80` が 4 種のメトリクスを section 数の関数として出力すること。

### Phase 0 結果の判定基準

| 観察 | シナリオ | 次の phase |
|---|---|---|
| Hausdorff > 0.1mm at $N=20$ | S1 | Phase 1 へ |
| Hausdorff < 0.1mm but normal angle > 1° somewhere | S2 | Phase 1 へ (より精密な local tally で TBR 分布を計測) |
| すべてのメトリクスが $N=10$ で既に収束 | S4 | 早期 pivot 検討 (section a-d) |

## 6. 中長期の展開 (案 1 を起点に何が広がるか)

案 1 が成功したあとの自然な拡張:

- **案 3 (UV coupling) を補論セクション化**: Interpolate API には UV 座標が組み込み済みなので、Phase 2 の終盤で「物理量 → UV map」を 1 例だけ追加すれば、案 3 の主張も同じ論文に乗せられる。または続報として独立論文化
- **案 2 (微分可能 CAD) への移行**: Interpolate API は制御点線形性を持っているので、`∂Solid/∂P_{ij}` を返す経路は技術的には自然な拡張。adjoint OpenMC は別問題なので surrogate model 経由から始める
- **案 4 (review)**: 案 1 の後に書けば、案 1 の結果を含めて open problem list を権威的にまとめられる

これらは案 1 を完了させてから決める。今は固定しない。
